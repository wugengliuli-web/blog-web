JavaScript重塑(6)——Event Loop

Posted on 2019-11-20 | In JavaScript Restore   |   字数统计: 2.2k(字)   |   阅读时长: 7(分) | Visitors

前言

  • 本篇内容主要是重塑JavaScript的第六篇，从JavaScript这门语言在宿主环境下的运行时机制，理解Event Loop，然后学习解决异步的实现。
我们一般都说，JavaScript是一个单线程、非阻塞、异步、解释性脚本语言。那么我们来一个个理解一下。

单线程

JS本身没有线程一说的，它的指向线程取决于它的运行环境，在常用的 浏览器 和 node 的宿主环境下，是单线程的运行机制。那么单线程是什么？

去排队买过火车票吗？假设只有一个窗口，每个人来了，只能按顺序排队，然后轮到了去购票，这就是单线程了。到了JS，就是一次只能执行一段代码，它不具备并行任务处理的特性。就是不具备同时开启多个窗口去卖票。

那么JavaScript的主线程是怎么推进这些事的呢？

非阻塞

好了，解释完了单线程调用栈的问题，我想，我们遇到了一个更大的问题，就是代码的“阻塞”，不过幸好，JS是一个“非阻塞”的语言；好了，解释下非阻塞吧。我们先不讨论“非阻塞”，先确认一下，什么是阻塞？

回到之前的那个例子，假如火车站排队卖票，只有一个窗口，那么后面的人越来越多，是不是贼慢，严重影响了大家的出行。
程序也是如此，我们一行行的执行代码，后面的代码不就要等前面的执行完成后才能执行，如果代码执行的很慢，售票员是个实习生，那不就阻塞的更加严重。

我们日常的一个简单打印，或许很快就出来，但假设我们打印从 1-1亿…嗯，确实很慢，还是别去操作了。
这个打印操作可能有点…瓜皮，那么，我们 加载图片、进行一个网络请求呢？我想，浏览器可不能把自己代码的执行速度建立在别人的响应速度上吧。
看一个阻塞的例子：

请求 xxx.js
请求 zzz.css
请求 ooo.png
count: 0

在这个例子中，我使用while模拟了一个同步的文件加载过程，点击按钮run，将会触发这个过程，可以打开浏览器控制台查看一下，按顺序加载这三个文件，这个run按钮可别多按啊，老哥们。问题在哪呢，加载的过程中，再去点击count+1按钮，它毫无反应，等到控制台现实三个文件都加载完毕，好家伙，刚才点的几次count全加上了。
这下我们明白了，当调用栈里面有东西的时候，你再去干别的，可以，排着队，想渲染页面，想运行代码，好的，等着吧。

“阻塞”我们搞清楚了，这是很糟糕的事情，严重影响了体验。那么“非阻塞”，很简单，就是解决掉这个问题就是“非阻塞”啦。
这就轮到我们的“异步”登场了！

异步

前面提到了，如果我们在加载文件或者进行一个网络请求，如果是同步进行，那就会导致“阻塞”的问题，那么解决这个问题的方法是“异步”。

具体怎么做呢？举个例子：
在一个老餐馆里，就是那种需要人分别去记录客人点单的餐馆，只有一个服务员(单线程)，此时来了三桌客人(三个函数)，如果按照同步思维，第一桌点完，才能轮到第二桌，那么后面的客人就想要个蛋炒饭，遇到第一桌的选择恐惧症得多难受鸭。此时的第一桌、第二桌、第三桌都是一个网络请求，快慢不同，但是不相互关联。
如何处理，为了不失礼貌，第一桌进来，“好的，客人想吃什么？您做下看看。”第二桌来了，同样，“好的，客人想吃什么？您做下看看。”，第三桌也是如此，当任意一桌想好了想吃啥了之后，唤一声“老板，来碗 腊肠蛋炒饭～”，即可完成点单。当然了，某一桌进来就想好了要吃啥，直接和老板讲就好啦。

换到我们代码执行里，那我们不就可以在执行到类似网络请求的时候，把它扔一边，我去干我的事，等你好了再叫我？
没错，这样好像解决了问题了。想想我们常用的一些API，setTimeout、setInterval、Promise等等，如果在Node的环境下就更多了，它们都有一个回调函数，等到一定的时间后才执行。

ok，异步的大概思路搞定了，就是我们把需要等待后在执行的函数给扔一边，继续我们的主线程，等它可以执行内容了在执行。但是这样还有点问题：

  1. 我们把等待的东西放一边去，放哪呢？
  2. 如果需要等待的事情有点多，它们又同时结束，那咋办，先执行哪一个？
  3. 有个等待的结束了，回来找主线程执行了，主线程正在做的事就要停下来做你的吗？

WebAPIs和任务队列

解决上面的几个问题，这就是WebAPIs和任务队列(Task Queue)的事啦。

  1. 正常的进行调用栈的操作，函数入栈，但是遇到异步任务时，放到一个由C++实现的浏览器创建的WebAPIs的线程中去完成，然后继续执行自己的同步任务，完成后出栈；
  2. 在主线程进行的过程中，WebAPIs会完成这些等待，把等待好的回调函数，放到“任务队列”中去；
  3. 这个时候有一个叫做Event Loop的角色，它在上面这几步的基础上，做一件事，监听调用栈是任务队列，当调用栈为空时，并且任务队列中有东西，那么就把任务队列中的第一项弹出，推入到调用栈中。调用栈则继续着第一步操作。
