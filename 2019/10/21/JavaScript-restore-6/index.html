JavaScript重塑(6)——Event Loop

Posted on 2019-10-21 | In JavaScript Restore   |   字数统计: 4.6k(字)   |   阅读时长: 16(分) | Visitors

前言

  • 本篇内容主要是重塑JavaScript的第六篇，从JavaScript这门语言在宿主环境下的运行时机制，理解Event Loop，以及研究任务与微任务。

我们一般都说，JavaScript是一个单线程、非阻塞、异步的脚本语言。那么我们来一个个理解一下。

单线程

JS本身没有线程一说的，它的线程取决于它的运行环境，在常用的 浏览器 和 node 的宿主环境下，是单线程的运行机制。那么单线程是什么？

去排队买过火车票吗？假设只有一个窗口，每个人来了，只能按顺序排队，然后轮到了去购票，这就是单线程了。到了JS，就是一次只能执行一段代码，它不具备并行任务处理的特性。就是不具备同时开启多个窗口去卖票。

那么JavaScript的主线程是怎么推进这些事的呢？

非阻塞

好了，解释完了单线程调用栈的问题，我想，我们遇到了一个更大的问题，就是代码的“阻塞”，不过幸好，JS是一个“非阻塞”的语言；嗯，那就解释下非阻塞吧。我们先不讨论“非阻塞”，先确认一下，什么是阻塞？

回到之前的那个例子，假如火车站排队卖票，只有一个窗口，那么后面的人越来越多，是不是贼慢，严重影响了大家的出行。
程序也是如此，我们一行行的执行代码，后面的代码不就要等前面的执行完成后才能执行，如果代码执行的很慢，售票员是个实习生，那不就阻塞的更加严重。

我们日常的一个简单打印，或许很快就出来，但假设我们打印从 1-1亿…嗯，确实很慢，还是别去操作了。
这个打印操作可能有点…瓜皮，那么，我们 加载图片、进行一个网络请求呢？我想，浏览器可不能把自己代码的执行速度建立在别人的响应速度上吧。
看一个阻塞的例子：

请求 xxx.js
请求 zzz.css
请求 ooo.png
count: 0

在这个例子中，我使用while模拟了一个同步的文件加载过程，点击按钮run，将会触发这个过程，可以打开浏览器控制台查看一下，按顺序加载这三个文件，这个run按钮可别多按啊，老哥们。问题在哪呢，加载的过程中，再去点击count+1按钮，它毫无反应，等到控制台现实三个文件都加载完毕，好家伙，刚才点的几次count全加上了。
这下我们明白了，当调用栈里面有东西的时候，你再去干别的，可以，排着队，想渲染页面，想运行代码，好的，等着吧。

“阻塞”我们搞清楚了，这是很糟糕的事情，严重影响了体验。那么“非阻塞”，很简单，就是解决掉这个问题就是“非阻塞”啦。
这就轮到我们的“异步”登场了！

异步

前面提到了，如果我们在加载文件或者进行一个网络请求，如果是同步进行，那就会导致“阻塞”的问题，那么解决这个问题的方法是“异步”。

具体怎么做呢？举个例子：
在一个老餐馆里，就是那种需要人分别去记录客人点单的餐馆，只有一个服务员(单线程)，此时来了三桌客人(三个函数)，如果按照同步思维，第一桌点完，才能轮到第二桌，那么后面的客人就想要个蛋炒饭，遇到第一桌的选择恐惧症得多难受鸭。此时的第一桌、第二桌、第三桌都是一个网络请求，快慢不同，但是不相互关联。
如何处理，为了不失礼貌，第一桌进来，“好的，客人想吃什么？您坐下看看菜单。”第二桌来了，同样，“好的，客人想吃什么？您做下看看。”，第三桌也是如此，当任意一桌想好了想吃啥了之后，唤一声“老板，来碗 腊肠蛋炒饭～”，即可完成点单。当然了，某一桌进来就想好了要吃啥，直接和老板讲就好啦。

换到我们代码执行里，那我们不就可以在执行到类似网络请求的时候，把它扔一边，我去干我的事，等你好了再叫我。
没错，这样好像解决了问题了。想想我们常用的一些API，setTimeout、setInterval、Promise等等，如果在Node的环境下就更多了，它们都有一个回调函数，等到一定的时间后才执行。

ok，异步的大概思路搞定了，就是我们把需要等待后再执行的函数给扔一边，继续我们的主线程，等它可以执行内容了再执行。但是这样还有点问题：

  1. 我们把等待的东西放一边去，放哪呢？
  2. 如果需要等待的事情有点多，它们又同时结束，那咋办，先执行哪一个？
  3. 有个等待的结束了，回来找主线程执行了，主线程正在做的事就要停下来做你的吗？

WebAPIs和任务队列

解决上面的几个问题，这就是WebAPIs和任务队列(Task Queue)的事啦。我们先建立一个模型：

  1. 正常的进行调用栈的操作，函数入栈，但是遇到异步任务时，放到一个由C++实现的浏览器创建的WebAPIs的线程中去完成，然后继续执行自己的同步任务，完成后出栈；
  2. 在主线程进行的过程中，WebAPIs会完成这些等待，把等待好的回调函数，放到“任务队列”中去；
  3. 这个时候有一个叫做Event Loop的角色，它在上面这几步的基础上，做一件事，监听调用栈和任务队列，当调用栈为空时，并且任务队列中有东西，那么就把任务队列中的第一项弹出，推入到调用栈中。调用栈则继续着第一步操作。

  7. Event Loop发现任务队列有了一个任务，并且调用栈为空，把cb函数弹出，推入到调用栈执行；
  8. console.log('wait');入栈，执行，然后出栈；
  9. cb函数执行完毕，cb出栈。

这个设计模型实际上还是蛮简单的。至此，一个事件循环的模型，我们已经可以清晰的表达了。

任务(Task)与微任务(MicroTask)

理解了单线程，非阻塞，异步的含义，也搞明白了Event Loop是在情况的基础上做什么样的一件事情。那么我们再回过头关注一下细节。

刚才讲到异步的时候，我们是想解决带有同步阻塞性质的问题，最终是通过异步回调机制解决的，将常用的一些带有延时性回调的 API 丢给WebAPIs去执行。完成后放入任务队列等待执行。这实际上是浏览器帮我们干的事，WebAPIs是浏览器创建的一个线程，setTimeout、setInterval、XHR这些 API 也是由宿主环境提供的，v8中找不到相关代码的。

本质上是，JavaScript就是一个单线程语言，一步步执行，浏览器添加了一些带有延时执行性质的API来处理需要I/O的特殊场景，然后利用Event Loop机制维护着单线程执行，以至于不会造成严重阻塞问题。

通常我们把经历过事件循环的任务称之为一个task，最终在任务队列等待执行，当然，我们的脚本实际上就是一个任务；但是如果我们想要添加一些在每个任务后面就执行的任务呢？
这个时候增加了一个概念叫做微任务(MicroTask)。
举例说明微任务队列：

同样还是在饭馆，只有一个服务员。来了三桌客人，第二桌客人想好吃啥了，把服务员叫去了，正常的点菜点完了，但是客人叫住服务员，给他们桌的人倒点茶水，拿点茶杯什么的。此时，第一桌，第三桌的人想好吃啥了，也还是等着服务员忙完了再点单。

在这个里面，三桌客人都是三个回调函数，第二个回调函数先完成，进入了调用栈，点完菜执行完成后，又想要点茶水，服务员给他们准备茶水，这就是微任务。等这些微任务执行完成后再继续为其他客人点单。

微任务也是一些异步执行的回调组成的队列，每一个任务（task）都有这样的一个队列，执行完任务，就会去查询是否有微任务需要执行，执行完成才能执行下一个task。

  2. 执行setTimeout入栈，把timeFn的函数信息和0ms延迟传递给WebAPIS去执行；setTimeout函数出栈；

如何区分任务与微任务

现在搞清楚了啥是微任务，但是看起来微任务和任务好相似哦，都是把回调放队列里，等待执行，只是执行的时机不同。那我们时机应用中如何辨别哪些是微任务，哪些是任务呢？

我们按照之前API的思路可以整理出：

Task就是经历过事件循环的异步事件，也可以说就是由宿主环境发起的任务。
MicroTask就是JavaScript引擎发起的，没有参与过事件循环的任务。

这里需要说明一点，微任务，看似也是一个异步函数，但实际上是在同步的机制上完成的，任务的主要工作是在别的线程里面完成然后放入任务队列等待，而微任务并没有在别的线程上执行，它只是在当前 JavaScript 代码执行完后立刻执行的，以此实现异步。

如何具体的区分他们，主要方式还是通过查看规范，这里直接做一下区分统计：

任务 chrome node 分类
IO yes yes Task
setTimeout yes yes Task
setInterval yes yes Task
setImmediate no yes Task
requestAnimationFrame yes no Task
process.nextTick no yes MicroTask
MutationObserver yes no MicroTask
Promise.then catch finally yes yes MicroTask

需要注意的是，这里是以chrome浏览器为主，在不同的宿主浏览器下实现可能会有不同，例如，有的浏览器会把Promise当作task来处理。

先看直接点击元素，点击后结果会出现在结果框中，它的处理顺序是这样的：

  1. 触发$inner的click事件回调，创建一个Task，然后将其推入调用栈；
  2. 触发$inner的click事件回调会引起冒泡，再将$outer的click事件回调添加进任务队列；
  3. 执行console.log('click')，推入栈执行完毕后出栈；打印'click'；
  4. Promise.resolve()入栈执行后出栈，将thenFn放入当前微任务队列；
  5. setTimeout入栈执行后出栈，将timeFn和0ms放入WebAPIS；开始计时；
  6. 修改$outer属性，触发MutationObserver微任务；将其放入微任务队列；
  7. 执行完当前任务，清空微任务，分别把thenFn和MutationObserver入栈后执行出栈；打印'promise'和'observer'；
  8. 此时调用栈和微任务队列均为空，任务队列中有$outer的click、timeFn，将$outer的click推入栈执行；
  9. 执行内容与添加和$inner的click一样；会输出'click'、'promise'和'observer'；并添加任务timeFn；
  10. 此时任务队列中还有两个timeFn，按照执行最终输出两个'timeout'。

如果使用按钮直接调用$inner的click方法，那么这里的触发不再是把$inner的click当作回调异步处理，而是一种dispatchEvent，相当于同步。这种方式相当于把该函数的调用作为点击该按钮时的一个上下文，其中一个执行栈而已，而其对应触发的冒泡，也会在这个上下文中执行，因此，会先输出两次'click'打印，然后查询微任务，而这里的'observer'打印一次是因为MutationObserver的触发在短时间内只触发一次，或者说在一个微队列中只触发一次。
