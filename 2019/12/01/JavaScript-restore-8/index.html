JavaScript重塑(8)——执行上下文

Posted on 2019-12-01 | In JavaScript Restore   |   字数统计: 5k(字)   |   阅读时长: 19(分) | Visitors

前言

  • 本篇内容主要是重塑JavaScript的第八篇，本篇内容在JavaScript运行时的基础上，细化知识点，研究代码执行时所做的事情及包含的信息，去理解我们平时讲的一些作用域、闭包、this等概念。

前面关于事件循环的那一篇，讲到的是 JavaScript 任务的运行机制，而一个任务中可能包含很多的函数调用等其他内容，在形成一个个任务，接下来，先入手的内容叫做“执行上下文”。通过“执行上下文”理解代码执行时包含的信息。

执行上下文（Execution Context）

在ECMASscript中的代码被分为三种类型，global、function和eval。每一种代码的执行都依赖自己的上下文。他们的特点是：

  • global: 全局环境，JavaScript运行便会首先进入该环境，其中可能会包括很多的function和eval的实例。
  • function: 函数环境，函数的每次调用都会进入这个函数执行的上下文。
  • eval: 运算执行环境，eval函数的每一次执行，也会进入eval执行中的上下文。

满满全是打印的一个例子，最终会顺序输出0-4，那么我们来大体描述一下它的执行：

  1. 代码执行首先进入了全局环境，记录了一些声明的变量；
  2. 执行console.log(0)，完成后，执行foo函数调用，进入一个函数环境foo；全局环境停止执行；
  3. 执行console.log(1)，执行完后，执行bar函数调用，进入一个另一个函数环境bar；foo函数环境也停止执行；
  4. 执行console.log(2)，完成后，函数bar已经执行完毕，返回；foo环境代码继续执行；
  5. 执行console.log(3)，执行完后，foo环境执行完毕，返回；全局环境继续执行；
  6. 执行console.log(4)，完成后，代码全部执行完毕。

嗯，这个流程大家肯定都知道，console.log()其实也是一个执行上下文就不细说了。这样细节的描述可以理清楚一点：
代码执行会先进入一个全局环境，而其中会发生很多调用，进入一些函数环境，而函数环境又可能会调用其它函数甚至是递归，生成更多函数环境，以此可能产生无限多个上下文环境。而当某一个函数环境执行完成后就会返回继续执行之前调用它的环境。这种一层层的调用实现的方式就是“栈”了。表现就是在事件循环那一篇讲的“调用栈”。

那么说了半天，什么是执行上下文呢？上面所关注的点好像也都是代码的执行。其实，执行上下文就可以理解为执行对应栈所需要的信息。
我们把执行上下文先抽象的理解为一个object，也就是键值对集合，那么每一个执行上下文都会保存一些属性来追踪关联代码的执行进度。
按照ES6的分类方式它主要包括三种主要信息：

  • Lexical Environment 词法环境
  • Variable Environment 变量环境
  • this value this指针

接下来来详细了解一下。

Lexical Environment

词法环境，简单来讲，可以抽象的理解为一个对象，记录的是“标识符”和具体函数及变量的映射关系。它实际存储的是 let,const 定义的函数声明和变量绑定。

一个词法环境一般由两部分组成：

  1. 环境记录(enviroment record)，用来存储变量和函数声明的具体位置。而它又根据执行环境类型不同，而分为两种。

    • 函数执行环境及catch中环境记录器为声明式环境记录器，存储变量、函数、和函数参数，声明式环境记录器包含一个传递给函数的argument对象。
    • 全局执行环境及with的执行环境的环境记录器为对象环境记录器。

  2. 对外部词法环境的引用(outer)，通过它访问父级的词法环境，和作用域相关。

this value

this指针，就是我们常说的this值，在这里先不讨论什么什么执行时this指向谁，刚才说了this是执行上下文的一个属性，它没有var this，只是在上下文创建阶段就确定了this的值。当我们在代码中使用了this，这个this的值就直接从执行的上下文中获取，而不是通过词法环境去搜索。this的值只取决中进入上下文时的情况。

作用域链(Scope Chains)

通过上面的执行上下文的创建，我们可以构思出一段代码在其创建时以及进入函数上下文的结构，通过这个结构，我们先知道一个概念叫做“作用域”。
在创建执行上下文的时候，把每一个变量/函数声明都保存在了环境记录（Enviroment Record）当中，而在执行的时候当使用到了某一个“标识符”，那么就会建立这个查找，通过Enviroment Record中的内容，找到对应的引用。而每一个Enviroment Record中保存的绑定关系就是其作用域。我们可以通过它确定一个变量所存在的周期以及区域。

重点来了，接着执行函数b中的内容，x = 5，在变量环境中有x设置为5，执行a()，在当前Enviroment Record下未找到a，通过outer，向外查找。查找GlobalExectionContext即全局上下文，发现函数a，开始执行，建立函数a的上下文，由于其函数a的声明所在上下文为GlobalExectionContext，所以将其设置为函数上下文的outer。

接着执行函数a，执行console.log(x)，在当前Enviroment Record下未找到x，根据outer向外查找，在全局环境下查找到x，且值为10，将其输出。

看了上面代码上下文的创建执行过程，作用域链其实就是在查找变量时，如果当前上下文中没有该变量，则根据outer向外查找，直到outer为null。因此，我们可以明白，为什么在不同的函数中声明的变量不一定能够访问得到，并且也清楚了变量的查找方式是和原型链相似的查找流程。

this在JavaScript中是一个关键字，执行的是当前代码块的运行环境，为什么需要这么一个东西呢？
我们知道JavaScript是没有类的概念的，也就是ES6才出现了class，我们有的是对象和函数，函数可以作为对象的一个方法来进行调用，也可以直接调用：

在前面简单的讲了this是执行上下文的一个属性，它是在执行上下文创建也就是函数调用的时候就被确定的，所以也是不可修改的。
在这里主要还是按照全局执行上下文和函数执行上下文两种类型来讲述。

构造函数的this

构造函数的this指向，这个问题，我们先要搞清楚什么是构造函数。其实就是通过new调用的就是构造函数。所以构造函数的this也可说就是new去调用this时的指向。
前面讲过了new做的事情，这里再看一下：

  1. new运算符接受一个构造函数和一组调用参数；
  2. 首先会在内存中开辟一块空间，创造一个空对象；
  3. 将对象的__proto__指向构造函数的原型；
  4. 将构造函数中的this指向该对象，并传入参数执行；
  5. 如果执行结果是对象类型，则返回该结果，否则返回上面获取的对象。
