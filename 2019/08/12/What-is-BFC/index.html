CSS-BFC

前言

  • 目的：平时经常使用BFC处理清除浮动影响，解决外边距塌陷等，但是对于原理一直掌握的不够牢靠，进行一下总结。
  • 内容：BFC是什么 => 如何形成BFC => BFC的特性及应用场景 => 总结

BFC是什么

  1. 定位方案
    在css2.1中，一个盒子有三种定位方案。
  • 普通流：一个萝卜一个坑，元素按着在html中的顺序来，块级box从上到下依次排列，两个box直接的垂直距离由box的垂直margin计算得到，是行内元素就和其他元素共处一行直到占满一行再换行。除另外指定，否则所有元素默认都是普通流定位。
  • 浮动：在浮动模型中，首先根据普通流布置盒子，然后根据浮动的方向尽可能向左或向右移动。内容可能沿着浮动的元素的一侧围绕排序(如文本环绕)。
  • 绝对定位流：元素完全脱离普通流，也就是说它与之后的兄弟姐妹元素再无瓜葛，并且按照指定的位置进行定位。

  1. Formatting Context
    格式化上下文相当于页面的一个渲染区域，有自己特定的渲染规则，决定子元素如何定位，以及与其它元素的关系和相互作用。

    Boxes in the normal flow belong to a formatting context.

    w3c文档中写到，普通流中的一个盒子都属于格式化上下文。我的理解是普通流排序的一个盒子，父级都有一个格式化的上下文，对这个子元素进行规则渲染。而这个构成格式化上下文的容器可能是它上一级格式化上下文中的普通流，也或是浮动、绝对定位流。

  2. BFC
    BFC就是一种属性，触发了这个属性的容器生成了一种特殊的格式化上下文，块级格式化上下文。在这个容器中，子元素都是以普通流进行渲染。容器内的所有元素可以说是一个独立的上下文关系，无论在里面做什么都不会影响到容器外的元素结构。并且容器本身有一些普通容器所没有的一些特性。

如何触发BFC

不同的格式化上下文需要特定的方式去触发，在css2.1中，有这么一段话：

Floats, absolutely positioned elements, block containers (such as inline-blocks, table-cells, and table-captions) that are not block boxes, and block boxes with 'overflow' other than 'visible' (except when that value has been propagated to the viewport) establish new block formatting contexts for their contents.

按照这个规则，可以得出触发BFC的条件是：

  1. 浮动元素：float 除 none 以外的值
  2. 绝对定位元素：position (absolute、fixed)
  3. display 为 inline-block、table-cells、table-captions中的任意一个
  4. overflow 除了 visible 以外的值 (hidden、auto、scroll)
  5. 以及我们的根元素html，是最大的一个BFC容器

BFC的特性及应用场景

BFC不仅有自己的渲染规则，而且会有一些自我特性，通过这些点可以灵活运用于实际开发中。

特定的子元素排序对齐规则

w3c中有规范

In a block formatting context, each box’s left outer edge touches the left edge of the containing block (for right-to-left formatting, right edges touch). This is true even in the presence of floats (although a box’s line boxes may shrink due to the floats), unless the box establishes a new block formatting context (in which case the box itself may become narrower due to the floats)

解决外边距合并

什么是外边距塌陷

外边距塌陷，在块级元素的上外边距和下外边距有时会合并为一个外边距，通常取值于其中的最大者，这种情况就是外边距塌陷。浮动元素和绝对定位元素的外边距不会折叠。一般发生在一下三种情况:

  1. 两个相邻的外边距会被折叠，上面一个块元素的 margin-bottom 和下面一个元素的 margin-top 会被折叠成其中的最大者。
  1. 如果父元素与第一个子元素之间，不存在上边框、上内边距、行内内容、这个父元素也没有创建BFC，或者清除浮动将两者的 margin-top 分开；或者在父元素与其最后一个子元素之间不存在下边框、下边距、行内内容height、min-height、max-height(不足该子元素展示时)将两者的 margin-bottom 分开，那么这两对外边距之间会产生折叠。此时子元素的外边距会 “溢出” 到父元素的外面。
  1. 如果存在一个空的块级元素，里面没有边框、内边距、行内内容分割 margin-top 和 margin-bottom，那么这个 margin-top 和 margin-bottom 会产生合并，这个合并的效果需要通过它的相邻元素位置去发现。

margin collapse的存在并不是一个bug，是为了解决p标签自带的上下margin值造成的视觉上的较大间隔问题。但是也造成了不符合逻辑的结果。

文本环绕的场景及原因

在BFC子元素排序中有浮动元素覆盖问题，这也是造成文本环绕的场景及原因。

上图就是一个浮动盒子周边文本环绕的效果，个人分析一下造成这个结果的原因：

  • 首先，最外层的container是一个BFC，其内的元素应该按照普通流进行排序，拥有BFC自己的对其规则。子元素的margin-box的左边缘与包含块left-padding-box的右边相接触。
  • 按照css规范中的定位方案，浮动元素先按照普通流进行排序，然后根据浮动的方向尽可能向左或向右移动，这里是左浮动，所以向左靠。浮动元素脱离文档流后，虽然不占据普通流的空间，但是会占据普通流中文本的空间。
  • 浮动元素排序完成，子元素p，要保证对齐规则，所以p元素不会移动仍在浮动元素下方。而浮动元素仍然占据普通流中的文本空间，所以形成了文字围绕着浮动元素存在的场景。

总结

  1. 触发BFC的方法有很多，本篇主要是说明BFC的特性和应用场景，使用的都是 overflow: hidde; 其他触发 BFC 的属性也可以达成效果，但是需要注意，每一个属性都带有自身的副作用，使用时需要考量一下场景。
  2. BFC就是一个独立容器，包裹的子元素无论在里面做什么都不会影响到容器外的元素，反之亦然。
