前言

  • 该系列内容主要是记录实现一个简易react实现的过程和思维，从而加深对React本身的理解。

本篇是实现简易React的第一篇，这是个一步步实现的过程，实现的过程就是发现问题和解决问题，每个实现的功能点也是由浅入深。并且注重的是功能实现，对于性能不多考虑。我把我的这一个小工具称之为Fishtail。使用的语法为TypeScript，使代码阅读起来更容易理解。

  1. ReactDOM.render基本功能，使用该方法，把一个React Element元素渲染到目标元素中。

  3. 实现基本Virtual DOM效果，利用render方法和JSX实现重(chóng)用DOM更新。

确定Instance结构

在React中，这种方式被称为 Reconciliation，意思是协调、和解。那么我们来想一想整个过程，如果我们要差异化更新，必须具备下面几点：

  1. 保存当前页面DOM结构映射出来的Fishtail Element结构树，因为这个是便于操作的，用来和最新生成的Fishtail Element结构树对比。
  2. 最新生成的Fishtail Element结构树是由再次调用render后通过createElement解析生成。
  3. 保存当前页面的DOM结构树，用于复用DOM，如果不复用，还有啥意义。
  4. 做差异判断，对属性和元素的增、删、改，还要考虑子元素的对比情况。
  5. 拿到最新生成的DOM结构树，添加到目标节点中。

按照这个思路，我们需要一个新的结构，这个结构要完成上面的1、3、4、5条对数据的要求，这个新结构称之为Instance，这个结构包含一下属性

  1. element：保存的是Fishtail Element结构树，这个可以通过每次render调用后进行保存即可。
  2. dom：保存页面的DOM结构树，用于复用渲染。
  3. childInstances：子元素Instance数组，用来对比子元素节点。

  2. Instance创建，会先创建A，然后递归创建找到B，再找到D和E，再继续找到C和F；

假设这一段结构被修改，删除了D，加入了G。

会产生的对比效果如图：

我们来细分一下这个DOM更新过程：

  1. 对比根节点，两边type相同，都为A，递归对比A的children；
  2. 对比A的第一个子节点，两边type相同，都为B。递归对比B的children；
  3. 对比B的第一个子节点，Instance这边的type为D，element的type为E，不同但是都存在，用E替换D；
  4. 对比B的第二个子节点，Instance这边的type为E，element这边不存在，删除E。B子节点对比完成；
  5. 回到A的第二个子节点对比，两边type相同，都为C。递归对比C的children；
  6. 对比C的第一个子节点，两边type相同，都为F。F没有子节点；继续对比C的子节点；
  7. 对比C的第二个子节点，Instance不存在，element这边为G，创建G的DOM并做添加。

通过上面的这个过程分析，最终想要更新成新的DOM结构，需要修改三个地方，分别在第3、4、7步。整体的流程是从根节点开始，深度遍历，一旦发现不相同的，就会按照reconcile方法中的规则进行增删改。如果发现某一个节点无法复用后，其子节点也不会再做对比，不是替换就是添加删除整个子节点。

总结

本篇内容，从实现DOM渲染，到解析JSX生成DOM，最后通过对比VDOM差异化更新页面。实现了React的几个基本功能点，但是可以发现，每次render都是在对比整个虚拟DOM树，并且每次更改都要显示的调用render，如果定义状态也是全局的。这些都可以利用本篇没有涉及到的组件来解决，下一篇将完成组件相关的功能点。
